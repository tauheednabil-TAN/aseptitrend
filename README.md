# AseptiTrend

*A portfolio project by [Tauheed Nabil](https://github.com/tauheednabil-TAN) — demonstrating a working, domain-accurate grasp of pharmaceutical environmental-monitoring trending, built to take the manual, repetitive parts of EM trending off a team's plate.*

**AseptiTrend is a proactive environmental-monitoring (EM) trending and excursion assistant for pharmaceutical aseptic filling.** It ingests viable (settle plate, contact plate, active air) and non-viable airborne particle results from classified cleanrooms (EU GMP Grades A–D), sets data-driven alert and action limits from historical baselines, caps them against the EU GMP Annex 1 (2022) specification ceilings, and continuously screens the current monitoring window for excursions and adverse trends — including the slow upward drift a static spreadsheet tends to hide. For any flag it can draft a first-pass investigation note to hand to a human investigator.

> **Scope & guardrails.** This is a **portfolio proof-of-concept running entirely on synthetic data**. It is **not validated, not GMP-qualified**, and it **never makes a batch, release, disposition, or quality decision**. Every output is decision-support for a **human-in-the-loop** who reviews and decides. The AI-drafted notes are explicitly labelled "for human review only." These constraints are surfaced throughout the product, not just in this file.

---

## Why it exists

EM trending is often manual, repetitive, and spreadsheet-bound: an analyst re-derives limits, eyeballs charts location-by-location, and can miss a gradual drift that never quite trips a single-point limit until it breaches. AseptiTrend takes that repetitive first pass off the team's plate — computing limits, screening every series against the applicable rules, and ranking what needs a human's attention by severity — while keeping the human firmly in control of every decision.

## Features

- **Dashboard / overview** — open flags by severity, a severity-ranked list of active excursions and adverse trends, and a per-grade compliance snapshot.
- **Trend view** — per (location, sample type) time series with **alert**, **action**, and **Annex 1 specification** limit lines drawn in; excursion points highlighted; the baseline-vs-monitoring boundary marked; hover for the exact value, date, and which limits applied.
- **Flags & excursions** — append-only excursion records (an audit-trail mindset) with full context, filterable by severity.
- **AI investigation draft** — a server-side Google Gemini call turns a single flag into a structured, clearly-labelled draft investigation note (summary, areas to review, immediate checks) framed as prompts for a human — never a conclusion or disposition.
- **Data** — view the dataset, regenerate the deterministic synthetic data, or import your own results by CSV (validated and normalised on import).

## Domain rules implemented

- **Grades & sample types** — A/B/C/D; viable settle plate (90 mm, 4 h), contact plate (55 mm), active air (CFU/m³); non-viable particles at ≥0.5 µm and ≥5 µm (particles/m³).
- **EU GMP Annex 1 (2022) reference limits** are hardcoded in [`src/lib/annex1.ts`](src/lib/annex1.ts) as the regulatory ceiling. Computed alert/action limits are always capped **strictly below** the applicable Annex 1 maximum; if a data-driven limit meets or exceeds spec it is clamped and flagged ("data-driven limit exceeds spec — spec governs").
- **Statistical limits** — alert = 90th percentile, action = 95th percentile of the historical baseline, recomputed whenever data changes. A **minimum of 20 baseline points** is required before a data-driven limit is trusted; otherwise it is labelled *provisional* and falls back to the Annex 1 ceiling.
- **Low-count guardrail** — where viable counts sit near zero (typical of Grade A/B), percentiles are unreliable, so the Annex 1 ceiling is adopted as the action limit (standard EM practice) rather than a spurious "action limit of 2 CFU".
- **Grade A** is treated as "no growth expected": any viable recovery ≥ 1 CFU is an immediate action-level event regardless of statistics.
- **Excursion & adverse-trend rules** — (1) action breach, (2) alert breach, (3) consecutive-rise trend (3+ non-decreasing results climbing toward the action limit), (4) repeated alerts in a rolling 7-sample window, (5) Grade A viable presence. See [`src/lib/flags.ts`](src/lib/flags.ts).
- **Baseline vs monitoring** — limits are derived from the historical baseline and excursion screening is applied to the current monitoring window, mirroring how EM programmes qualify limits and then apply them (rather than retroactively flagging the very data that defined the limit).

## Synthetic data

The bundled dataset ([`src/data/seed.json`](src/data/seed.json), generated by [`src/lib/generateData.ts`](src/lib/generateData.ts)) is seeded and reproducible: ~90 days across 8 locations spanning all four grades and multiple sample types, mostly compliant, with three deliberately injected storylines so the tool has something to find:

1. **Gradual drift → action breach** at *Compounding Suite — Grade C* (settle plate).
2. **Isolated Grade A viable hit** at *Filling Line 1 — Grade A (Point FA-1)* (active air).
3. **Repeated alert-level results** at *Equipment Prep — Grade C* (contact plate).

CSV import expects the header `date,room,grade,sample_type,value,unit` and rejects malformed rows with row-numbered messages.

## Tech stack

Next.js (App Router) + TypeScript · React · Tailwind CSS · Recharts · Google Gemini API (server-side only) · Vitest. No database — the app boots from the bundled seed and holds data in memory.

## Run locally

```bash
npm install
npm run dev        # http://localhost:3000
```

Optional — enable AI investigation drafts:

```bash
cp .env.example .env.local
# set GEMINI_API_KEY=... in .env.local  (get one from https://aistudio.google.com/apikey)
```

Without a key the AI feature degrades gracefully to an "unavailable" state; everything else works out of the box on the bundled data.

```bash
npm test           # Vitest — statistics, Annex 1 capping, all 5 flag rules, CSV, storylines
npm run build      # production build
npm run seed       # regenerate src/data/seed.json
```

## Environment variables

| Variable         | Required | Purpose                                                        |
| ---------------- | -------- | -------------------------------------------------------------- |
| `GEMINI_API_KEY` | No       | Enables AI investigation drafts. Read **server-side only**.    |
| `GEMINI_MODEL`   | No       | Override the Gemini model (defaults to `gemini-2.0-flash`).    |

Secrets live only in environment variables and are never committed; `.env*` files are git-ignored.
